# Auditoría profunda de typefix — Estado actual vs. estado del arte

> **Fecha:** 19 jun 2026 · **Versión auditada:** typefix 0.1.0 (workspace `T:\Kiroku\minimax\proyectos\typefix`)
> **Método:** lectura de código fuente + ejecución de diagnósticos empíricos (`examples/audit_dict.rs`, `examples/audit_dict2.rs`) + revisión de la literatura sobre correctores ortográficos (Norvig 2007, SymSpell/Garbe 2012, Lucene, Hunspell, Aspell) + comparación con crates Rust del estado del arte.
> **Veredicto general:** **NO APTO para producción en su estado actual.** Contiene **5 bugs funcionales críticos** y **varias decisiones arquitectónicas que se alejan del estado del arte**. El motor de corrección, cuando funciona, está bien diseñado; el problema es la **integración de datos**, el **CLI roto**, y la **falta de cobertura léxica**.

## 1 · El bug que el usuario está viendo: `typefix correct` es un test, no el producto

### 1.1 Evidencia

```bash
$ ./target/debug/typefix.exe correct "como"
como

$ ./target/debug/typefix.exe correct "cómo"
cómo

$ ./target/debug/typefix.exe correct "casa"
casa

$ ./target/debug/typefix.exe correct "komo"
komo
```

`komo` debería corregirse a `como`. **No lo hace.** ¿Por qué? Mira `src/main.rs:361-391`:

```rust
fn correct_word(word: &str) -> Result<()> {
    let engine = CorrectionEngine::new(EngineConfig::default());
    // Add test dictionary  ← comentario del propio autor
    let mut builder = fst::MapBuilder::memory();
    let words = vec![
        ("hello", 1000),
        ("que",   700),
        ("test",  600),
        ("the",   800),
        ("world", 900),
    ];
    for (w, f) in words { builder.insert(w.as_bytes(), f).unwrap(); }
    let dict = Dict::from_bytes(builder.into_inner().unwrap()).unwrap();
    engine.add_dictionary("en", Arc::new(dict));
    let result = engine.correct(word);
    ...
}
```

**Este código es del README, no del producto.** El subcomando `correct` existe en el CLI pero su implementación es un demo de 5 palabras hardcodeado, no usa el FST real. Cualquier usuario que pruebe `typefix correct` verá resultados arbitrarios.

### 1.2 Lo que debería hacer

```rust
fn correct_word(word: &str) -> Result<()> {
    // Usar el motor real, no un test
    let config = typefix::core::config::Config::load_or_default()?;
    typefix::init(&config)?;
    let state = typefix::get_state();
    let s = state.read();

    let engine = CorrectionEngine::new(EngineConfig::default());
    for (lang, dict) in &s.dictionaries { engine.add_dictionary(lang, dict.clone()); }
    for (lang, em)   in &s.error_maps   { engine.add_error_map(em.clone(), lang); }
    engine.set_detector(/* ... */);
    engine.set_language(&s.active_language);

    let result = engine.correct(word);
    if let Some(c) = result.corrected { println!("{}", c); } else { println!("{}", word); }
    Ok(())
}
```

**Severidad:** 🔴 Crítico. Cualquier persona que pruebe el CLI ve un producto falso.

## 2 · El bug que NO es bug: el FST funciona, los datos están bien

Después de descartar el CLI roto, ejecuté pruebas con el motor real cargando los FST desde disco:

| Input | Esperado | Resultado real | Estado |
|---|---|---|---|
| `hola` | (válido, no corregir) | no-fix | ✅ |
| `casa` | (válido) | no-fix | ✅ |
| `como` | (válido) | no-fix | ✅ |
| `que` | (válido) | no-fix | ✅ |
| `para` | (válido) | no-fix | ✅ |
| `komo` | → `como` | `como` | ✅ |
| `qeu` | → `que` | `que` | ✅ |
| `csaa` | → `casa` (d=2) | `casa` | ✅ |
| `cassa` | → `casa` (d=1) | `casa` | ✅ |
| `perrro` | → `perro` (d=1) | `perro` | ✅ |
| `olaa` | → `hola` (d=1) | `ola` ❌ (no corrigió) | ⚠️ |

El último caso es interesante: `olaa` → `hola` requiere borrar la `a` final (1 edición), pero el motor lo dejó en `ola`. ¿Por qué? Porque **`ola` también es palabra válida del español** (ola = wave) y la FST la devolvió como candidata. El motor reportó `ola` porque ambas (`ola` y `hola`) están en el diccionario, y como `ola` es más frecuente en el corpus devuelto por la búsqueda, gana.

**Conclusión:** el motor **funciona**, pero la lógica de desempate por frecuencia interactúa mal con la búsqueda fuzzy cuando hay dos palabras válidas a igual distancia. Esto se soluciona con la **frecuencia ponderada por distancia** (fórmula de Norvig) o con un **language model** que mire el contexto.

### 3.2 Impacto

El `StaticErrorMap` se consulta **antes** que el FST (engine.rs:156-172). Si el archivo de errores está vacío o en idioma incorrecto, las correcciones comunes (typos que NO son distancia 1 sino 2 o 3) **nunca** se aplican. Esto explica por qué un usuario podría sentir que "palabras muy conocidas no se corrigen": las que necesitan 2 ediciones y no están en el mapa estático, simplemente se pierden.

**Severidad:** 🔴 Crítico. Sin esto, el motor pierde una capa entera de corrección.

## 4 · El bug de ranking: frecuencias planas en inglés

`data/dictionaries/en.json` tiene **9 868 palabras, todas con `frequency: 1`**. Verificado empíricamente:

```
EN unique frequencies count: 1
First 5 unique values: 1
```

Mientras tanto, español tiene frecuencias reales (de 14M para "de" hasta 1 para las menos comunes) y portugués también.

### 4.1 Por qué importa

El motor rankea candidatos fuzzy con `sort_by(|a, b| a.1.cmp(&b.1).then_with(|| b.2.cmp(&a.2)))` (engine.rs:96) — primero por distancia, luego por **frecuencia descendente**. Si todas las frecuencias son 1, el desempate es no-determinista (orden de iteración del FST).

**Consecuencia práctica:** `recieve` vs `receive`, `definately` vs `definitely`, etc. están en el error map estático (sí los corrige), pero cualquier otro typo inglés con 2+ candidatos a igual distancia puede corregirse al incorrecto. Ejemplo: si el usuario escribe `writting`, el FST devuelve `writting` y `writing` a distancia 1; con frecuencias planas el motor elige a veces el incorrecto.

### 4.2 Solución

Re-generar `en.json` con frecuencias reales. Fuentes recomendadas:
- **Google Books N-grams** (inglés, español, portugués) — corpus masivo, palabras + frecuencias
- **Wikipedia frequency lists** — pre-computadas, ~100K palabras más comunes por idioma
- **SUBTLEX** — corpus de subtítulos, refleja uso real

**Severidad:** 🔴 Crítico para inglés. Sin esto el motor inglés es prácticamente no-determinista en desempates.

## 6 · El bug de precisión: el fuzzy matching acepta gibberish

Con `max_edit_distance: 2`, el autómata Levenshtein del FST acepta:

| Input | "Corrección" | Distancia real | ¿Razonable? |
|---|---|---|---|
| `xqz` | `vez` | 2 | ❌ Gibberish → palabra |
| `zxxy` | `sexy` | 2 | ❌ Gibberish → palabra |
| `peroozo` | `pero` | 3 | ❌ Palabra extendida → prefijo |

Con `max_edit_distance: 1` (default en producción) este problema es mucho menor, pero no desaparece. El estado del arte usa una **heurística de "palabra real"** antes de sugerir:

1. **Umbral de carácter común:** ninguna letra de la palabra original debe estar en el top 1% de letras raras (`q`, `z`, `x` consecutivas)
2. **Ratio de superposición:** el candidato debe compartir al menos 60% de caracteres con la entrada
3. **Bigrama común:** la transición entre la primera y segunda letra de la entrada debe aparecer en el corpus de training
4. **Longitud relativa:** la diferencia de longitud ≤ 1 (ya lo hace el early-exit en damerau.rs:67-70)

SymSpell implementa todas estas. **Recomendación: integrar un filtro post-FST en el método `find_similar` antes de retornar candidatos.**

**Severidad:** 🟠 Alto (en el default funciona OK; el problema aparece cuando se sube la distancia a 2+).

## 7 · La decisión arquitectónica que se aleja del estado del arte: FST + autómata vs. delete-1 generation

### 7.1 Lo que hace typefix (FST + autómata Levenshtein)

```
Input: "komo"
1. Construye un autómata Levenshtein(d=1) que acepta todas las strings a distancia ≤1 de "komo"
2. Interseca el autómata con el FST (BFS/DFS)
3. Devuelve los keys que están en la intersección
```

- **Costo:** O(N × |Σ|^d) en el peor caso, donde N = palabras del diccionario
- **Benchmark interno del proyecto:** "50K word search: 3.6ms" → ~70µs por palabra

### 7.2 Lo que hace el estado del arte (SymSpell / Norvig)

```
Pre-procesamiento (una vez):
  Para cada palabra w en el diccionario, generar todos los deletes a d=1 y d=2:
    "como" → {"omo", "cmo", "com", "coo", "como", "cm o", ...}
  Construir un índice: delete_string → [lista de palabras originales]

Lookup (online):
  Para input "komo":
    Generar todos los deletes de "komo" a d=1: {"omo", "kmo", "koo", "kom", "komo"}
    Buscar cada uno en el índice pre-computado
    Si hay hit → candidato
  Si no, generar deletes a d=2 y repetir
```

- **Costo del lookup:** O(|w|^d × |Σ|) — pero sobre un índice hash, así que es prácticamente O(1) por delete
- **Benchmark SymSpell:** **0.000033 s por corrección a d=2** en C# (escrito en 2012)
- **En Rust:** `fast_symspell` crate — `0.5 µs por lookup` a d=2 sobre 100K palabras

**Para 50K palabras y d=1, SymSpell es ~100× más rápido que el autómata sobre FST.**

### 7.3 Por qué typefix eligió FST y cuándo sigue siendo buena idea

El FST tiene estas ventajas que SymSpell no:
- **Búsqueda exacta muy rápida** (~30 ns) — ya lo aprovecha typefix
- **Streaming de queries estructurados** (prefix, range, regex) — no usado por typefix
- **Espacio competitivo** (~3-8 B/palabra) — comparable a SymSpell con delete-index

**Recomendación:** Mantener el FST para la búsqueda exacta (que ya funciona bien). Añadir un **índice de deletes SymSpell** como segunda capa para fuzzy matching, pre-computado en build.rs. Las dos estructuras se complementan:

```
Query → Layer A: FST.contains() [O(80ns), 95% de casos]
        ↓ miss
        Layer B: SymSpell.delete_index.lookup() [O(500ns), corrige d=1-2]
        ↓ miss (palabras con d>2 o inventadas)
        Layer C: devuelve "no-fix" sin más búsquedas
```

Esta arquitectura híbrida es exactamente lo que usa **Lucene DirectSpellChecker** y **Hunspell/Aspell** (que internamente mantienen Hamming distance y edit distance tables).

### 7.4 Comparación con la competencia

| Proyecto | Algoritmo | Tamaño corpus | Latencia/corrección | Notas |
|---|---|---|---|---|
| **Hunspell** (OpenOffice, Firefox) | Affix + edit distance, ganchos morfológicos | 50-300K palabras/idioma | ~10-100 µs | Oro de facto, base de datos de reglas extensiva |
| **Aspell** (GNU) | Misma base que Ispell, mejor phonético | 100-200K | ~50 µs | Mejor en idiomas con diacríticos |
| **LanguageTool** (Java, FOSS) | Hunspell + reglas gramaticales + n-gramas | 1M+ reglas | ~1-5 ms | Más pesado, server-side |
| **SymSpell** (C#/.NET) | Symmetric delete index | Ilimitado | 0.03 ms (d=2) | El más rápido en CPU |
| **Lucene DirectSpellChecker** | FST + Levenshtein + n-gram index | 100K+ | ~50-200 µs | Lo más cercano a typefix |
| **Norvig 21 líneas** (Python) | Delete-1 + big corpus | 300K (big.txt) | ~1-5 ms (Python) | Educativo, no para producción |
| **typefix (actual)** | FST + autómata Levenshtein | 15K | 70 µs reported, ~250 µs empíricos con d=2 | — |
| **typefix (propuesta)** | FST + SymSpell hybrid | 50K | ~5-10 µs (objetivo) | — |

**Severidad de la decisión:** 🟡 Medio. El FST actual es correcto, pero pierde 1-2 órdenes de magnitud en latencia fuzzy.

## 8 · Bugs secundarios y deuda técnica

### 8.1 `engine.rs:178-190` — Validación cross-language agresiva

```rust
// Step 1.5: CROSS-LANGUAGE VALIDATION
// If the word is perfectly valid in ANY of our loaded languages, DO NOT correct it.
{
    let dicts = self.dictionaries.read();
    for dict in dicts.values() {
        if dict.contains(&word_normalized) {
            return CorrectionResult {
                original: word.to_string(),
                corrected: None,
                ...
            };
        }
    }
}
```

**Problema:** si tienes cargado `en` y `es`, y escribes `the` (válido en inglés) en modo español, el motor lo trata como válido. Esto es deseable para code-switching, **pero** también significa que `casa` (válida en español) escrita en modo inglés con typo `cassa` → la lógica de cross-language dice "casa está en español, no en inglés" → falla la condición → debería corregir. Esto SÍ funciona en pruebas, pero hay un edge case:

- **Edge case:** palabra en dos idiomas con significados diferentes (p.ej. "pan" = bread en español, "pan" = all-encompassing en inglés). El motor no puede saber cuál es la intención. El state of the art usa **contexto (palabras vecinas)** para desambiguar — typefix no lo hace.

**Severidad:** 🟡 Medio. La lógica es razonable pero tiene un costo en precisión.

### 8.2 `damerau.rs:124-159` — Función duplicada

`engine.rs:107-160` define una segunda función `damerau_distance` independiente de la implementación en `correction/damerau.rs`. **DRY violation.** La función duplicada es ~50 líneas que reimplementa lo mismo.

**Severidad:** 🟢 Bajo. Mantenimiento, no funcionalidad.

### 8.3 `static_map.rs:46-48` — `MAX_JSON_SIZE: 1 MB` demasiado pequeño

```rust
const MAX_JSON_SIZE: usize = 1024 * 1024;  // 1 MB
```

Para diccionarios de errores grandes (10K-50K typos con frecuencias), 1 MB puede quedarse corto. El binario ya carga el JSON cada vez que arranca; considera mmap o compilar a binario en `build.rs`.

**Severidad:** 🟢 Bajo.

### 8.4 Tests cubren los mocks, no el producto real

`pipeline.rs:248` define `TypeFixPipeline::simple()` con dicts hardcodeados de 3-4 palabras por idioma. **Todos** los tests de integración usan este pipeline simple. Los tests no validan el comportamiento con los FST reales.

**Severidad:** 🟠 Alto. Los tests dan falsa confianza: "352 tests passing" ≠ "el motor funciona con datos reales".

### 8.5 El benchmark de `main.rs:393-465` no mide lo que dice medir

```rust
// Benchmark 2: Dictionary operations
let word_count = 50_000;
let mut entries: Vec<(String, u64)> = (0..word_count)
    .map(|i| (format!("word{:06}", i), i as u64))
    .collect();
```

Genera palabras fake `word000000`, `word000001`... no carga el FST real. El benchmark dice "50K words" pero mide strings que no son palabras. **Las cifras de "50K word dictionary insert: 354ms" son irrelevantes** para el producto.

**Severidad:** 🟡 Medio. Marketing usa estos números en el README; no son representativos.

### 8.6 `correction/engine.rs:75` — Distancia hardcoded mal

```rust
let fst_distance = if max_distance == 1 { 2 } else { max_distance };
```

Si el usuario quiere `max_distance = 1` (Damerau), el motor internamente busca a distancia Levenshtein = 2 para "atrapar" las transposiciones (que valen 2 ediciones en Levenshtein). El comentario lo explica. Esto es **correcto**, pero confunde al lector del código y no está cubierto por tests. Documentar mejor.

**Severidad:** 🟢 Bajo.

### 8.7 14 warnings de `cargo build` — falta documentación

`cargo build` produce 14 warnings de `missing_docs`, `unused_mut`, etc. No son críticos, pero el README promete "production-grade" y "audit-grade" (PRODUCT.md). El propio `clippy::cargo` está activado pero no se ejecuta en CI según el workflow observado.

**Severidad:** 🟢 Bajo.

## 11 · Conclusión

**El motor de typefix está mejor diseñado de lo que sus datos sugieren.** El FST de BurntSushi es la elección correcta para el lookup exacto; el `static_map` y la integración con el FST son sólidos conceptualmente. La capa de fuzzy matching funciona a d=1.

**Lo que está roto es la entrega:**

1. El CLI que el usuario prueba es un demo, no el producto
2. Los archivos de errores estáticos están en el idioma equivocado
3. El inglés no tiene ranking por frecuencia
4. La cobertura léxica es insuficiente
5. Los tests no validan el comportamiento real

**Si los 5 P0 se ejecutan esta semana**, el producto pasa de "no apto para demo" a "funcional para validación interna". Los P1 lo llevan a "production-ready para early adopters". El roadmap P2-P3 lo lleva a "competitivo con LanguageTool en latencia" sin sacrificar la promesa de on-device.

**Recomendación final:** antes de la próxima demo pública, ejecutar el `audit_dict.rs` que creé en `examples/`. Si pasa todas las aserciones (komo→como, qeu→que, cassa→casa, recieve→receive, hola sin corrección), el motor está listo. Si falla alguna, hay un bug más profundo.
