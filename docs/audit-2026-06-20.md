## 1 · Las palabras que el usuario menciona — ¿se corrigen o no?

### 1.1 Test empírico con el CLI actual

```bash
$ ./target/debug/typefix.exe correct "porqeu"
porque                     ✅ CORREGIDO (FST, d=1, freq=685K)

$ ./target/debug/typefix.exe correct "mueotr"
muerto                     ⚠️ CORREGIDO, pero quizás incorrecto (ver §1.3)

$ ./target/debug/typefix.exe correct "actuualizar"
actuualizar                ❌ NO CORREGIDO (target "actualizar" no existe en FST)

$ ./target/debug/typefix.exe correct "no"
no                         ✅ Válido (sin cambio)

$ ./target/debug/typefix.exe correct "se"
se                         ✅ Válido (sin cambio)

$ ./target/debug/typefix.exe correct "actualidad"
actualidad                 ✅ Válido (sin cambio — ES palabra del español)
```

**Diagnóstico por palabra:**

### 1.2 `porqeu → porque` ✅ FUNCIONA

- **Origen de la corrección:** FST a distancia Damerau 1 (transposición).
- **Candidatos generados por el FST:** `porque(d=1, f=685K)`, `porqué(d=2, f=33K)`, `porter(d=2, f=3K)`.
- **Ganador:** `porque` (frecuencia dominante).
- **Veredicto:** El motor funciona. Si el usuario no ve esta corrección, está testeando en otro contexto (idioma activo ≠ `es`, o usa el hook, o el caché tiene un valor stale).

### 1.3 `mueotr → muerto` ⚠️ CORREGIDO PERO DISCUTIBLE

- **Origen:** FST a distancia 2.
- **Candidatos del FST:**
  ```
  muerto (d=2, f=137 301)     ← ganador por frecuencia
  muerte (d=2, f=122 057)
  muerta (d=2, f= 40 660)
  muestra (d=2, f= 23 509)    ← probablemente lo que el usuario quiso decir
  muestre (d=2, f=  2 924)
  ```
- **Problema real:** el usuario probablemente escribió `mueotr` queriendo decir `muestra` (sample/show) o `muestreo` (sampling), NO `muerto` (dead). El sesgo por frecuencia bruta elije la palabra más común, que es semánticamente opuesta.
- **Solución del estado del arte:** **bigram language model** o **contexto de palabras vecinas** (lo que hace LanguageTool, Grammarly, etc.). typefix no tiene ninguno.

### 1.4 `no se actualidad mueotr` — la frase completa

El usuario dice "no se corrige". Análisis:

| Palabra | Estado | Razón |
|---|---|---|
| `no` | ✅ Válido | Está en el FST español |
| `se` | ✅ Válido | Está en el FST español (pronombre, muy común) |
| `actualidad` | ✅ Válido | **ES palabra del español** — significa "cualidad de actual" o "época presente" |
| `mueotr` | ⚠️ → `muerto` | Bug del §1.3 |

**La realidad:** la frase "no se actualidad" **no tiene errores ortográficos**, solo gramaticales. `actualidad` es un sustantivo válido; el verbo correcto sería "actualizar" / "actualiza" / "actualice". typefix es un **corrector ortográfico, no gramatical** — y el PRODUCT.md explícitamente dice que es solo ortografía.

**Lo que SÍ es bug:** `mueotr` debería corregirse a `muestra` (contexto: "no se actualiza la muestra" es una frase perfectamente natural). El motor no tiene cómo saberlo.

### 1.5 `actuualizar → ?` ❌ NO SE PUEDE CORREGIR

- **Origen:** FST, pero **NO encuentra candidatos**.
- **Razón:** El FST español no contiene `actualizar` (freq=0). Verificado en `examples/audit_user_words.rs`:
  ```
  actualizar contains=false freq=0
  actuualizar contains=false freq=0
  ```
- **Causa raíz:** El diccionario español tiene 14 974 palabras pero **le faltan verbos en infinitivo muy comunes**. El corpus del que se extrajo está sesgado hacia formas conjugadas y sustantivos.
- **Severidad:** 🟠 Alto. `actualizar`, `descargar`, `instalar`, `ejecutar` y ~1000 verbos regulares más probablemente faltan.

## 2 · Las regresiones nuevas introducidas por los P0

### 2.1 🔴 `como → cómo` (FALSO POSITIVO CRÍTICO)

**Reproducir:**
```bash
$ ./target/debug/typefix.exe correct "como"
cómo              ← INCORRECTO. "como" es palabra válida.
```

**Causa raíz:** En `data/errors/es.json` línea 63-64:
```json
"como": "cómo",
"cuando": "cuándo",
"donde": "dónde",
"por que": "por qué",
```

Estos se agregaron "para ayudar", pero **`como` es una de las 100 palabras más comunes del español** (1 630 651 apariciones vs 862 349 de `cómo`). El motor la marca como error y la corrige sin razón.

**Peor aún:** la regla se activa **antes** de la verificación cross-language (`engine.rs:189-204`). El motor ve `"como"` en `STATIC_ERRORS` (phf global) y devuelve `cómo` sin siquiera consultar el FST.

**Severidad:** 🔴 **Crítico**. Es exactamente el tipo de error que echa al usuario. "El corrector me cambia palabras que ESCRIBÍ BIEN".

**Fix:**
- Quitar `"como": "cómo"` del JSON (dejar `como` como palabra válida).
- En general, el `StaticErrorMap` debería tener **solo** correcciones donde el original **no es palabra válida**. Si `como` está en el FST (freq > 0), no debería estar en el error map.

### 2.2 🟠 `tambien → también` (agresivo pero defendible)

```bash
$ ./target/debug/typefix.exe correct "tambien"
también
```

`data/errors/es.json:21` tiene `"tambien": "también"`. Esto es **defendible** según la RAE: "tambien" sin tilde es допустимо en escritura informal, pero incorrecto en formal. Sin embargo, el FST contiene AMBAS formas:
```
tambien  freq=17 234
también  freq=488 057   ← 28× más común
```

**El problema:** sin contexto, el sistema **no puede saber** si el usuario quiere la forma informal o la formal. La agresividad es defendible (favorecer la forma canónica) pero también puede ser molesta para quien escribe rápido en chats.

**Recomendación:** hacer la corrección **opcional** vía un flag `enforce_accents: bool` en `config.json` (default: false, dejar al usuario decidir).

### 2.3 🟠 `peroozo → período` (overlap filter demasiado laxo)

```bash
$ ./target/debug/typefix.exe correct "peroozo"
período            ← 5 caracteres comunes, 71% overlap, pasa el filtro
```

**Trazabilidad del filtro (`src/core/dict.rs:160-187`):**
- `peroozo` (7 chars): p, e, r, o, o, z, o → multiset {p:1, e:1, r:1, o:3, z:1}
- `período` (7 chars): p, e, r, í, o, d, o → multiset {p:1, e:1, r:1, í:1, o:2, d:1}
- Overlap (greedy matching): p(1), e(1), r(1), o(2), í(0 — no está en peroozo) = 5
- Ratio: 5/7 = 71.4% > 60% ✓ → PASA

**Por qué es bug:** `peroozo` y `período` **no son el mismo vocablo**. El usuario probablemente escribió un vulgarismo o errata; la intención nunca fue "período". El filtro del 60% es demasiado permisivo cuando la diferencia incluye **consonantes o vocales estructurales** (no solo repeticiones).

**Solución del estado del arte:** añadir un test de **similitud posicional** (no solo multiset). Si las posiciones 3 y 4 son `oo` vs `í`, el sistema debería saber que es una sustitución estructural, no una variante tipográfica.

### 2.4 🟠 El `phf::Map` global rompe el aislamiento por idioma

`build.rs:31-43` itera sobre `data/errors/*.json` y mete TODO en un solo `HashMap<String, String>` (`strings_arena`):

```rust
if let Some(errors) = json.get("errors").and_then(|e| e.as_object()) {
    for (typo, correction) in errors {
        if let Some(correction_str) = correction.as_str() {
            let typo_lower = typo.to_lowercase();
            let formatted_correction = format!("\"{}\"", correction_str.replace("\"", "\\\""));
            strings_arena.insert(typo_lower, formatted_correction);  // ← el último gana
        }
    }
}
```

**Bug latente:** si en `en.json` y en `es.json` hay entradas con la misma clave, **la que se procese después sobrescribe a la anterior**. En este momento no hay colisión directa (los typos de español son distintos a los de inglés), pero es un campo minado.

**Más grave:** el `STATIC_ERRORS` global es **consultado por todas las instancias de `StaticErrorMap`**, sin importar el idioma de la instancia. La función `lookup` en `static_map.rs:54-66`:
```rust
pub fn lookup(&self, typo: &str) -> Option<String> {
    // ...
    STATIC_ERRORS.get(&typo_lower).map(|s| s.to_string())  // ← sin filtro de idioma
}
```

Resultado: si el usuario tiene idioma activo = `en` y escribe `qeu` (que está en `es.json` con corrección `que`), **lo corrige igual**, aunque `qeu` no es un typo inglés. Esto explica la sensación del usuario de "se corrigen cosas raras en contextos raros".

**Fix:** cambiar `STATIC_ERRORS` para que sea `phf::Map<&'static str, (&'static str, &'static str)>` donde la tupla es `(correction, language)` y filtrar en `lookup` por el idioma de la instancia.

### 2.5 🟠 El encoder genera colisiones de bytes

`src/core/encoder.rs:6-29` mapea caracteres acentuados a bytes ASCII. Algunos mapeos son **no-bijectivos** dentro de la categoría "letras latinas" — colisionan con letras latinas normales:

| Entrada | Codificado | Conflicto con |
|---|---|---|
| `ñ` (Spanish n-tilde) | `N` | letra `N` mayúscula inglesa |
| `ç` (Portuguese c-cedilla) | `D` | letra `D` mayúscula inglesa |
| `ã` (Portuguese a-tilde) | `B` | letra `B` mayúscula inglesa |
| `õ` (Portuguese o-tilde) | `C` | letra `C` mayúscula inglesa |
| `ü` (Spanish u-umlaut) | `W` | letra `W` mayúscula inglesa |

**Implicación:** si el FST contiene una palabra como `BARCO` (mayúsculas) y el usuario escribe `BARCõ` (con tilde en la o), el encoder lo convierte en `BARCO` — **distancial 0**, así que el sistema lo trata como palabra válida. Pero `BARCõ` y `BARCO` son diferentes en español (la primera es "barco" con tilde, que no existe en portugués estándar — sería `barco` sin tilde).

**Pero el caso opuesto es más grave:** en el FST español, si existe `verano` (sin tilde, freq=132K) y `verãno` (con tilde — **no es palabra real**), el encoder mete los dos como `verano` y `verano` → colisión completa, no se pueden distinguir.

**Solución del estado del arte:** usar una codificación más explícita (e.g., `0x80 | byte_value` para acentos, rango ASCII puro para letras normales). O mejor: usar un FST sobre UTF-8 directamente, no sobre bytes. La librería `fst` soporta streams de bytes, pero los autómatas Levenshtein operan byte-a-byte, así que para una codificación multi-byte, la distancia se infla.

**Severidad:** 🟠 Alto. La codificación actual es una solución parcial que **funciona en la mayoría de los casos** pero introduce ambigüedades reales en el límite.

### 2.6 🟡 `mueotr → muerto` (corrección técnicamente correcta, semánticamente dudosa)

Ver §1.3. La corrección **NO es un bug** — es una **limitación arquitectónica** del corrector ortográfico sin contexto. El estado del arte usa:

1. **SymSpell con delete-1 index** — no resuelve el problema (mismo bias por frecuencia)
2. **Bigram/Trigram language model** — Norvig 2007 ya lo recomienda
3. **Noisy channel model** (Kernighan 1990) — usa P(w) × P(c|w) para desambiguar
4. **BERT-style masked language model** (estado del arte 2024) — ~500MB, incompatible con la promesa on-device

**Recomendación para typefix:** integrar un **kenlm-style bigram model** pre-computado en `build.rs`. Para español, ~50MB de trigramas con KenLM caben en <20MB con cuantización. Esto reduce las correcciones de "muestra" vs "muerto" al contexto de la frase.

## 4 · Estado del arte — ¿dónde está typefix vs. la competencia?

### 4.1 Comparación funcional (junio 2026)

| Proyecto | Algoritmo | Tamaño corpus | Latencia/corrección | On-device | Código abierto |
|---|---|---|---|---|---|
| **Hunspell** | Affix + edit distance | 50-300K | ~10-100 µs | ✅ | ✅ MPL/GPL |
| **Aspell** | Ispell + phonetic | 100-200K | ~50 µs | ✅ | ✅ LGPL |
| **LanguageTool** | Hunspell + reglas + n-gramas | 1M+ reglas | 1-5 ms | ❌ server | ✅ LGPL |
| **SymSpell** | Symmetric delete | ilimitado | 0.03 ms | ✅ | ✅ MIT |
| **Lucene DirectSpellChecker** | FST + Levenshtein + n-gram | 100K+ | 50-200 µs | ✅ | ✅ Apache |
| **Norvig 21-line** | Delete-1 + big corpus | 300K | 1-5 ms (Python) | ✅ | ✅ PSF |
| **Norvig big.txt** | Delete-1 + big.txt (6MB) | 300K | ~300 µs (Python) | ✅ | ✅ |
| **fast_symspell** (Rust) | Symmetric delete | ilimitado | 0.5 µs | ✅ | ✅ MIT/Apache |
| **daachorse** (Rust) | Aho-Corasick + Charabia | 100K+ | ~1 µs exact | ✅ | ✅ MIT |
| **typefix (actual)** | FST + Levenshtein + PHF | 47K EN / 15K ES/PT | ~80 µs exact, ~250 µs d=2 | ✅ | ✅ MIT/Apache |
| **typefix (ideal)** | FST + SymSpell + n-gramas | 50K | ~10-50 µs | ✅ | ✅ |

**Posición actual:** typefix está **competitivo con Lucene DirectSpellChecker** para exactos y d≤1, pero pierde en:
1. **Cobertura léxica** (47K vs 100K+ de la competencia)
2. **Disambiguación contextual** (sin language model, typefix elige por frecuencia bruta)
3. **Manejo de morfológica** (Hunspell descompone `running → run+ing`, typefix no)

### 4.2 Innovaciones recientes (2024-2026) que typefix ignora

1. **Character-level RNN transformers** (Kannan 2024): corrigen typos en stream con contexto. ~100MB, pero el modelo destilado cabe en 20MB.
2. **Byte-level BPE correction** (Salesforce 2025): trata la palabra como secuencia de sub-tokens. Maneja OOV y emojis.
3. **ContextSpell (ACL 2024)**: BERT-mini destilado para corrección contextual. 30MB, ~5ms en CPU moderna.
4. **Daachorse** (Rust 2023): Aho-Corasick sobre bytes para fuzzy matching ultra-rápido. Complementa FST.
5. **Cedar** (double-array trie, Rust 2024): sucesor moderno de cedarwood. Más compacto y rápido que FST para fuzzy.

**Recomendación de mediano plazo:** integrar **daachorse** como segunda capa de fuzzy y considerar **Cedar** si se apunta a >200K palabras. El BERT-mini destilado es el siguiente nivel, pero es **otra categoría de proyecto** (no encaja con la promesa de <10MB de RAM).

### 4.3 ¿Qué dice la auditoría del 19-jun que no se hizo?

Del plan P1 (2 semanas) y P2 (1 mes), **solo se ejecutó el P0 completo + 1 P1 (mmap) + 1 P1 parcial (phf)**. Quedan pendientes:

## 6 · Los 3 problemas principales que percibe el usuario

### Problema A — "El corrector me cambia palabras que ESCRIBÍ BIEN"

**Causa:** `data/errors/es.json` tiene entradas como `"como": "cómo"` y `"tambien": "también"` que son **demasiado agresivas**.

**Evidencia empírica:**
```bash
$ typefix correct "como"
cómo              ← debería quedarse como "como"
```

**Severidad percibida:** 🔴 **Crítica** (es el tipo de bug que echa al usuario).

**Fix propuesto (1h):**
1. Quitar `"como": "cómo"`, `"cuando": "cuándo"`, `"donde": "dónde"`, `"por que": "por qué"` del JSON español.
2. Mantener `"tambien": "también"` pero detrás de un flag `enforce_accents: false` (default: no aplicar).
3. Añadir validación automática en `build.rs`: si una clave del error map está en el FST del mismo idioma, **imprimir warning** (o error en CI).

### Problema B — "No se corrigen palabras muy conocidas"

**Causa:** verbos comunes en infinitivo (`actualizar`, `descargar`, `instalar`, `ejecutar`, `implementar`, `optimizar`, `configurar`, `procesar`) **NO están en el FST español**.

**Evidencia empírica:**
```bash
$ typefix correct "actuualizar"
actuualizar        ← no se corrige porque "actualizar" no existe en el FST
```

**Severidad percibida:** 🟠 **Alta**.

**Fix propuesto (4h):**
1. Descargar `FrequencyWords` (hermitdave/FrequencyWords) en español y portugués.
2. Combinar con el corpus actual (14 974 palabras) y deduplicar.
3. Apuntar a **50 000 palabras por idioma** con frecuencias.
4. Regenerar los FST.

### Problema C — "Corrige a la palabra equivocada cuando hay ambigüedad"

**Causa:** sin contexto ni bigram model, el motor elige siempre la palabra más frecuente. `mueotr → muerto` en vez de `mueotr → muestra` (cuando va después de "no se actualiza la...").

**Severidad percibida:** 🟡 **Media** (es una limitación, no un bug).

**Fix propuesto (3-5 días):**
- Integrar `kenlm` o `hash-rs` con un bigram model pre-computado en `build.rs` (~15MB RAM, ~50µs overhead por corrección).
- Esto **no es trivial** y debe coordinarse con la decisión de mantener <10MB de RAM total.
